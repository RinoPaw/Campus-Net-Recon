# 第四篇：破局——流量穿透与内网互联

在经历了对 ARP 代理的深度拆解、对 TTL 路由路径的实测以及对非对称路由的“左右互搏”后，我已经彻底摸清了校园网的底层骨架。

虽然通过手动绑定 `ip neigh` 可以实现零延迟的二层直连，但这种方式极不稳定：网关的 Proxy ARP 像个不知疲倦的猎手，一旦 ARP 缓存老化，流量就会立刻被劫持回机房。

为了实现**长期、稳定、且不占用额外名额**的访问方案，我决定从“硬刚”转向“借力”。

## 4.1 核心策略：应用层代理 (SOCKS5)
既然网关通过三层路由（IP 层）和二层劫持（ARP 层）来监控流量，那么我只需将流量封装在已认证的“合法通道”内，就能实现透明穿透。

**我的方案是：将手机作为电脑的“流量中转站”。**

### 实施步骤：
1. **手机端 (Android)**：运行 `Clash Meta for Android`，开启 `Allow LAN` (允许局域网连接) 功能。
2. **电脑端 (Windows/Linux)**：在系统代理设置中，将代理地址指向手机的局域网 IP（例如 `10.202.207.207:7890`）。
3. **流量封装**：电脑的所有公网流量通过 SOCKS5 协议发送给手机，手机再通过校园网认证通道发送给网关。

## 4.2 为什么这个方案能破局？
*   **身份伪装**：在网关眼中，所有流量都来自已登录的手机。电脑的 IP 消失了，网关无法通过 MAC 地址识别出这台电脑，自然也就无法触发“设备数超限”的报警。
*   **协议隔离**：代理流量在手机上被封装成了普通的 TCP/UDP 数据流，网关的深度包检测 (DPI) 无法轻易区分这是手机自己的流量，还是电脑转发过来的流量。
*   **稳定性**：不再需要手动绑定 ARP，避免了与网关的 Proxy ARP 抢答冲突，彻底告别了“丢包”和“延迟抖动”。

## 4.3 意外的惊喜：校内互联的“特权”
通过这个代理方案，我还意外获得了一个“特权”：**校内资源访问**。

因为手机本身就处于校园网的逻辑网段内，通过代理，电脑可以直接访问校内的图书馆数据库、教务系统，甚至是那些只有校内 IP 才能打开的实验服务器。这种体验比传统的 VPN 更加轻量，且完全不需要额外的账号权限。

## 4.4 结语：网络协议的博弈论
回望这段校园网逆向之旅，从最初对 `255.254.0.0` 掩码的困惑，到最终通过 `Clash` 实现流量穿透，我学到的不仅仅是网络工具的使用，更是一种思维方式：

**网络从来不是绝对的围墙，它是一套基于协议和策略的规则集。**

*   网关的 **Proxy ARP** 是规则，**ARP 缓存绑定** 是对抗；
*   **状态防火墙** 是规则，**流量封装 (代理)** 是绕过；
*   **IP 限制** 是规则，**地址伪装 (SNAT)** 是破局。。
