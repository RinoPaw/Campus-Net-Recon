# 02-锐捷网关的幻术

## 2.1 监听 ARP 流量
上篇提到，`nmap` 扫描的结果显示几乎所有 IP 都指向同一个 MAC 地址 `28:D0:F5:74:30:0E`。这显然是锐捷核心交换机在进行 ARP 代理（Proxy ARP）。为了确认这一点，我决定直接在链路层（Layer 2）进行监听。

我在 Android 手机的 Termux 中运行 `tcpdump`，直接抓取网卡上的 ARP 数据包：
```bash
sudo tcpdump -i wlan0 arp -n
```

屏幕上刷新的数据包显示，每当我尝试获取某个 IP 的 MAC 地址时，网关总是抢先返回它自己的 MAC 地址。但当我静默监听时，偶尔能抓到一些设备间的直接通信，这些包里包含着真实的 MAC 地址。这说明网关的代理并非无处不在，它只在检测到我发起 ARP 请求时，才会主动进行拦截和代答。

## 2.2 触发 NFPP 限速
为了摸清网关的底线，我使用 `arp-scan` 对网段进行了一次全量扫描。结果出现了明显的断层：

```text
10.203.95.0     28:d0:f5:74:30:0e       Ruijie Networks
... (中间 200 多个 IP 无回应) ...
10.203.95.241   28:d0:f5:74:30:0e       Ruijie Networks
```

这种“两头有、中间无”的现象，是锐捷网关开启了 NFPP（Network Foundation Protection Policy，网络基础保护策略）的结果。该机制通过令牌桶算法限制了每个用户在单位时间内的 ARP 请求配额。当我扫描速度过快时，配额耗尽，网关直接拒绝响应；待扫描停止、配额补充后，后续的 IP 又能被扫到了。

## 2.3 真实的物理边界：/21 网段
实验中我发现一个规律：在 `10.201.32.0` 到 `10.201.39.255` 这个范围内，我能扫到大量的真实 MAC 地址；一旦跨出这个范围，就只剩下网关的代理 MAC。

虽然学校下发的是 `/16` 的超大子网掩码，但物理上，我所在的区域被严格划分在一个 `/21` 的 VLAN（Virtual Local Area Network，虚拟局域网）中。在这个“水池”里，二层广播是互通的；而水池之外，则必须经过网关的三层路由转发。

## 2.4 为什么网关要这么做？
这种“全员代理”的设计，主要为了实现用户隔离：
1. **防止内网攻击**：网关强制接管所有 ARP 请求，使得设备间无法直接进行二层通信，阻断了 ARP 欺骗和内网蠕虫传播。
2. **流量审计**：强制所有流量经过网关，方便学校进行流量统计和计费。

这种设计虽然严密，但并非无懈可击。只要能绕过这些代理机制，就能在内网实现直接通信。
