# 第二篇：扒开锐捷网关的“幻术”——ARP 代理与 NFPP 机制

在上篇笔记中，我通过 `nmap` 扫描发现了一个令人沮丧的事实：几乎所有的 IP 都指向了同一个 MAC 地址 `28:D0:F5:74:30:0E`。这显然不是真实的设备，而是锐捷核心交换机在搞鬼。

为了探究这层“幻术”背后的真相，我决定从链路层（Layer 2）直接切入。

## 2.1 监听空气中的秘密
我不再使用高层扫描工具，而是直接调用了网络分析的“手术刀”——`tcpdump`。在 Termux 中，我输入了以下命令，试图监听所有的 ARP 流量：

```bash
sudo tcpdump -i wlan0 arp -n
```

屏幕上疯狂刷新的数据包让我大开眼界。我不仅看到了网关的“代答”，还意外地抓到了一些**从未见过的真实 MAC 地址**。

**关键发现**：
*   **网关的代答 (Proxy ARP)**：当我主动扫描时，网关总是抢在真实设备前回应。
*   **泄露的真实流量**：当我静默监听时，网络中偶尔会漏出一些设备间的直接对话。这些包里包含着完全不同的 MAC 地址，这证明了**网关的代理并非无处不在，它只是在“主动拦截”我的请求**。

## 2.2 触发“安全机制”：NFPP 限速
为了摸清网关的底线，我运行了 `arp-scan` 对网段进行全量扫描。结果出乎意料：

```text
10.203.95.0     28:d0:f5:74:30:0e       Ruijie Networks
... (中间 200 多个 IP 无回应) ...
10.203.95.241   28:d0:f5:74:30:0e       Ruijie Networks
```

扫描结果呈现出一种诡异的“两头有、中间无”的现象。经过分析，我意识到这是锐捷网关的 **NFPP (Network Foundation Protection Policy)** 机制在起作用：
1.  **令牌桶算法**：网关给每个用户分配了有限的 ARP 代理额度（约 30 个包）。
2.  **强制冷却**：当我快速扫描时，配额耗尽，网关直接“装死”拒绝回应。
3.  **自动恢复**：在扫描间隙，网关的令牌桶得到补充，于是结尾的 IP 又能被扫到了。

## 2.3 真实的物理边界：/21 网段
随着实验的深入，我发现了一个有趣的规律：在 `10.201.32.0` 到 `10.201.39.255` 这个范围内，我能扫到大量的真实 MAC；而一旦跨出这个范围，就只剩下那令人厌烦的锐捷代理 MAC。

**结论**：
虽然学校下发的是 `/16` 的超大掩码，但物理上，我所在的区域被严格划分在一个 **`/21` 的 VLAN（虚拟局域网）** 中。在这个“水池”里，二层广播是互通的；而水池之外，则必须经过网关的三层路由转发。

## 2.4 为什么网关要这么做？
我意识到，这种“全员代理”的设计，本质上是为了实现**“用户隔离”**：
*   **防止内网攻击**：网关强制接管所有 ARP 请求，使得学生设备之间无法直接进行二层通信，从而阻断了 ARP 欺骗和内网蠕虫病毒的传播。
*   **流量审计**：强行让所有流量经过网关，方便学校进行流量统计和计费。

然而，这种设计并非完美。只要我能绕过这些“保安”，我就能直接与室友的设备建立连接，实现真正的“内网穿透”。
