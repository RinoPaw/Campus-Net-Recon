# 第三篇：深潜——TTL 揭示的物理真相与路由逃逸

在前两篇笔记中，我通过 `nmap` 和 `tcpdump` 揭开了锐捷网关 Proxy ARP 的幻术，并摸清了校园网“逻辑上大一统，物理上碎成 `/21` VLAN”的底层拓扑。

但真正的挑战在于：**我如何绕过那个总是试图劫持流量的网关，实现两台设备之间的直接通信？**

## 3.1 测绘“物理距离”：TTL 的秘密
为了验证两台设备是否真的处于同一个物理二层网络，我祭出了网络排错的终极工具——`ping`。

我观察到一个有趣的现象：
*   **TTL=64**：Ping 某些设备时，TTL 保持为 64，且延迟极低（< 10ms）。
*   **TTL=63**：Ping 另一些设备时，TTL 变为 63，且延迟暴增至 60ms 以上。

根据 IP 协议，每经过一次三层路由，TTL 就会减 1。这说明：
*   **TTL=64**：数据包在二层（交换机/AP 内部）直接交换，未经过机房核心网关，这是真正的“直连”。
*   **TTL=63**：数据包被强制送入了机房的 AC（无线控制器）进行三层路由转发，这意味着我被“强制绕路”了。

我意识到，只要我能让流量保持在二层交换，就能彻底绕过网关的限速、审计和代理劫持。

## 3.2 极限操作：手动篡改 ARP 缓存
为了实现“二层直连”，我决定手动干预 Linux 内核的 ARP 缓存。我使用 `ip neigh` 命令，强行将目标设备的 IP 与其**真实的 MAC 地址**绑定，而不是那个网关的 MAC。

```bash
# 手动删除错误的网关绑定
ip neigh del 10.201.9.84 dev wlan0
# 强行绑定真实的 MAC 地址
ip neigh add 10.201.9.84 lladdr 86:82:8f:3a:fa:e4 dev wlan0 nud permanent
```

当我按下回车的那一刻，奇迹发生了：**Ping 的 TTL 瞬间从 63 变回了 64，丢包率降为 0%，延迟也回归到了毫秒级。**

我成功了！我绕过了机房那台昂贵的锐捷核心 AC，直接在宿舍天花板的 AP 内部实现了流量交换。

## 3.3 意料之外的“非对称路由”故障
实验过程中，我遭遇了一个极其诡异的现象：我修改了 A 手机的 ARP 表，但 A Ping B 时，回包的 TTL 依然是 63。

经过深入分析，我发现这是**非对称路由 (Asymmetric Routing)** 导致的：
1. **去程**：A 按照我指定的真实 MAC，直接发给了 B（二层）。
2. **回程**：B 的 ARP 缓存还没到期，它依然认为 A 的 IP 对应的是网关的 MAC，于是把回包发给了网关。
3. **网关的“政审”**：网关收到这个“莫名其妙”的回包，因为没见过对应的请求包，触发了防火墙策略，导致丢包。

这让我深刻理解了企业级网络的严密性：**在有状态防火墙 (Stateful Firewall) 的环境下，去程和回程必须走同一条路，否则就会被当成非法攻击而丢弃。**

## 3.4 结语：规则之上的博弈
通过这次深潜，我不仅证实了校园网 AP 并没有开启绝对的二层隔离，更看清了网关是如何通过 Proxy ARP 和状态防火墙来“锁死”流量的。

我们所谓的“上网”，其实是运行在这些规则之上的。而通过理解这些规则，我找到了在规则缝隙中穿行的路径。下一篇，我将分享如何利用这些发现，实现优雅的内网穿透与流量代理。
